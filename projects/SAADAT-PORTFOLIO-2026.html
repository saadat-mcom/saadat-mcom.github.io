Project 1 · Mastercom Australia · 2024–Present
Master Navigation System (MNS) — Enterprise Transit Route Intelligence Platform
Sole Developer: Solution Architecture · Android Mobile · Back-End · GTFS Processing · HERE Integration · Database · Geospatial · Performance Optimisation
120× faster load (60 s → 0.5 s) −80% server API calls 90%+ cache hit rate 5,000+ routes per feed NSW · VIC · QLD · WA · UK networks

Designed and delivered a full-stack enterprise transit navigation platform enabling public transport operators to transform static GTFS datasets into production-grade turn-by-turn navigation experiences. The platform bridges the gap between transport planning systems and operational vehicle navigation by converting GTFS routes, trips, stops and shape geometry into HERE route handles that support real-time navigation, simulation and driver guidance.

GTFS Feed Sources (NSW/VIC/QLD/WA/UK) ↓ Backend: Rails + PostgreSQL + PostGIS + Sidekiq ├── GTFS Processing Engine (route/trip/stop/shape) ├── Route Handle Generation (HERE Route Matching API) └── Task Processing + Async Workers ↓ HERE Platform Services ├── Routing API v8 ├── Route Matching API └── Route Handle API ↓ Android Navigation Client (Kotlin, HERE SDK 4.18.5) ├── Multi-region feed selection ├── Route & trip discovery (Room DB cache) ├── Live navigation (turn-by-turn, manoeuvre detection) ├── Route simulation (variable speed) └── Stop-arrival detection + deviation recovery

Key Technical Achievements

  • Multi-tier caching (Room DB + WorkManager + version-aware invalidation): reduced load time from 60 s to 0.5 s (120×), cut API calls by 80%
  • Clean Architecture: UI → Business Logic (NavigationCore, RouteCalculator) → Data Layer (GtfsRepository) → Infrastructure (Room, Retrofit2, HERE SDK)
  • HERE Route Handle generation engine: GTFS shape → SocketCAN trace → HERE Route Matching → HERE Route Handle → stored and served to Android client
  • Manoeuvre detection: RIGHT_FORK, RIGHT_U_TURN, ROUNDABOUT, complex intersections — visual + textual guidance
  • Route deviation detection and automatic re-routing with 50 m threshold and 5 s debounce
  • Route simulation engine with configurable speed (1× to N×), camera-tracking lock/unlock
  • Stop-arrival detection validates transit ops against scheduled stop sequences
  • ProGuard + resource shrinking: build size reduced from 2.0 GB → ~200 MB

Engineering Outcomes

  • Production transit navigation across Australian and UK networks
  • Supports bus drivers onboarding new routes with turn-by-turn guidance
  • Free-form point-to-point routing via geocoding for diversions
  • Offline-capable route browsing through Room DB persistence

Project 2 · Mastercom Australia · 2024–2025 · ~2 months (Sole Developer)
Fleet Telemetry & Edge Computing Platform — J1939/CAN Bus to Cloud
Sole Developer: Edge Runtime · AWS Cloud Ingest · DBC Decoding · Analytics · Deployment Tooling · Field Documentation
5,000–8,000 CAN frames / 5–6 s Full lifecycle ownership Vehicle bus → Edge → Cloud → Analytics Validated on real Scania bus MF4 recordings

Delivered an end-to-end fleet intelligence platform capable of collecting raw vehicle CAN Bus/J1939 traffic, enriching it with DBC signal metadata, filtering and batching at the edge, uploading to AWS cloud ingest, decoding and storing in OpenSearch for operational fleet analytics. Developed as a practical transport technology system (not a prototype): includes Raspberry Pi edge runtime, physical CAN hardware integration, local debugging dashboards, AWS Copilot deployment, OpenSearch indexing and customer-facing field documentation.

Vehicle J1939 CAN Bus (bus / heavy vehicle) → Passive CAN tap / CAN Crocodile / direct CAN interface → Raspberry Pi Edge Device (Python systemd service) ├── CanListener (SocketCAN can0, python-can) ├── J1939 Parser (29-bit arbitration ID → PGN, priority, SA) ├── FilterEngine (PGN allowlist/blocklist) ├── BatchManager (frame grouping) ├── Spool (atomic write, offline persistence) ├── Uploader (HTTPS POST, retry) ├── HeartbeatService (health, config version, queue depth) ├── ConfigManager (remote config, atomic persist) └── Watchdog (service health, CAN interface, upload freshness) → AWS Copilot Load-Balanced Python FastAPI (can-ingest) ├── DBC decode (cantools, j1939_base.dbc) ├── PGN/SA masking for multi-ECU support └── OpenSearch bulk index (can-frames-YYYY-MM-DD) → OpenSearch → Dashboards, fleet analytics, debugging

Edge Architecture Highlights

  • Python systemd service on Raspberry Pi OS — auto-boot, auto-recovery
  • Passive J1939 capture — no vehicle bus interference
  • DBC-driven cloud decoding keeps edge lightweight; DBC changes don't require re-deploying edge devices
  • Local spooling with atomic writes preserves data during network loss
  • Configurable PGN filtering reduces bandwidth — discovery mode or controlled forwarding
  • Remote config fetch/persist with atomic writes; filter updates without physical device access
  • Watchdog monitors CAN interface state and upload freshness for autonomous recovery

Cloud & Operational Highlights

  • AWS Copilot Load-Balanced Web Service (Python 3.12 Docker, FastAPI + Uvicorn)
  • Async handoff to thread pool — edge device gets fast 200 ACK, indexing failures logged separately
  • OpenSearch daily indices (can-frames-YYYY-MM-DD) for retention management
  • OpenSearch documents contain: device ID, PGN, DBC labels, raw hex payload, decoded signals
  • Automated Pi provisioning: SPI/CAN overlays, bitrate, Ethernet/DHCP/WiFi, service autostart
  • MF4 replay injector for lab validation using real Scania bus recordings
  • Customer setup guides for non-developer field staff

Project 6 · MST Global / Komatsu Australia · 2021–2024
Helix Virtual LightBar Device Platform
Architect & Lead Engineer: Device Virtualization · Go / Golang · Industrial IoT · REST APIs · Mining Safety Systems
100% hardware dependency eliminated N simultaneous virtual devices Zero-cost dev & demo environments Full PatLite protocol compatibility

Designed and built a software-based virtual device platform that emulates multiple physical LightBar safety devices simultaneously — compatible with existing Helix Dashboard workflows and PatLite communication interfaces. Physical LightBars in underground mining environments were scarce and expensive; this platform eliminated that dependency entirely, enabling development, integration testing and customer demonstrations without any physical hardware.

Helix Dashboard Layer ↓ (REST Control API + Status API) LightBar Dashboard Layer (configure · monitor · command) ↓ Virtual Device Platform (hardware abstraction layer — Go) ├── Virtual LightBar Instance 1 → localhost:10002 (independent state) ├── Virtual LightBar Instance 2 → localhost:10003 (independent state) ├── Virtual LightBar Instance 3 → localhost:10004 (independent state) └── Virtual LightBar Instance N → dynamically allocated Each instance: color · flash · alert · buzzer · operational status

Engineering Design

  • Single simulator instance spawns N independent virtual LightBars, each with unique addressable REST endpoint and isolated internal state
  • Control API: color changes, flash patterns, alert states, buzzer commands — identical interface to physical device
  • Status API: real-time state query matching physical LightBar response format
  • Config-driven: number of devices, IP address, starting port — instant environment setup
  • State model: current color, flash config, alert config, buzzer config, operational status
  • Visualization layer for live inspection of active colors, flash patterns, alert conditions

Project 7 · Mastercom Australia · 2024–2025
MP70 Fleet Reliability & Recovery Initiative
Lead: Root-Cause Analysis · Ruby on Rails · ALMS Integration · Reliability Engineering · Test Strategy
Root-cause resolution Safer automated fleet recovery Restored test coverage

Led a deep technical investigation into reliability issues affecting remotely deployed Sierra Wireless MP70 gateways and the AVTA application ecosystem. Gateways appearing online while telemetry had stopped flowing created operational blind spots; automated recovery was not consistently restoring service. Identified root causes, restored test coverage and defined a roadmap for safer, scalable fleet recovery automation.

Key Findings

  • Gateway online status ≠ application health ≠ telemetry health
  • Recovery actions were heavyweight starting with reinstall operations
  • No distinction between connectivity, application and telemetry failures
  • Vulnerable to concurrent/overlapping recovery attempts against the same gateway
  • Limited post-recovery business-level validation

Recommended Improvements

  • Gateway-level recovery locking
  • Recovery cooldown periods
  • Staged recovery: lightweight checks → full reinstall
  • Separate health assessments: gateway / application / telemetry
  • Post-recovery telemetry validation
  • Fleet-wide rate limiting for recovery operations
  • Expanded automated test coverage and regression validation
